# v3 設計仕様: PM 自律化レイヤ（GROOM フェーズ + 横断メモリ）

> ステータス: **ドラフト（スコープ合意済み・詳細設計は実装フェーズで詰める）**
> 作成: 2026-06-22
> 関連: [ロードマップ & スコープ（v2 / v3+）](https://linear.app/edereship/document/looppilot-os-ロードマップ-and-スコープv2-v3-3f38a59ae90d) / ES-437 E2E 検証での議論

## 0. 背景と目的

v2 では Codex(PM) が「次の1件を選定（A1）」「実装 brief を生成（A2）」するところまで実現した。しかし PM の行為はまだ受動的で、チケットの整理・優先度変更・新規起票・分割といった能動的な盤面管理は人間に依存している。

v3 の目的は **Codex を完全自律の PM に昇格**させ、「AI が自律的にプロダクトを開発・運用・改修まで実施する」最終ゴールへの基盤を作ること。

## 1. スコープ

### A3. PM アクション列（GROOM フェーズ）

Codex(PM) の出力を「1件選定 + 一行根拠」から**構造化されたアクション列**へ拡張し、オーケストレーターが検証・代理実行する。

#### 1.1 フェーズ配置

```
GROOM → SELECT → PLAN → IMPLEMENT → HANDOFF → MONITOR → DONE
```

- GROOM は SELECT の前に毎ループ実行する独立フェーズ
- PM がまず盤面を俯瞰して整理してから、SELECT で次の1件を選ぶ
- 将来的に GROOM の頻度や深さを独立にスケール可能

#### 1.2 アクション範囲（フル PM 権限）

| アクション | 説明 |
|-----------|------|
| `reprioritize` | 既存チケットの優先度を変更 |
| `update` | チケットの説明・タイトル・ラベルを更新 |
| `create` | ギャップ検出・QA・バグ発見から新チケットを起票 |
| `split` | 大きなチケットをサブタスクに分割 |
| `close` | 不要・重複チケットをクローズ |
| `label` | ラベルの追加・削除 |
| `update_memory` | 横断メモリの更新（B2 と統合） |

#### 1.3 実行方式

- Codex がアクション列を JSON で出力
- オーケストレーターが各アクションを検証・代理実行
- Linear API キーはオーケストレーター側で保持（Codex には渡さない = IPI 対策）
- 全アクションを監査ログ（DB）に記録

#### 1.4 人間ゲート

不要（ship-and-correct）。PM が判断して即実行、問題は事後修正。監査ログで人間が後から確認可能。

#### 1.5 入力コンテキスト

GROOM フェーズで PM に渡す情報:
- 要求 / 要件定義（B1 の specs/）
- 横断メモリ（B2 の docs/memory/）
- 全チケット盤面（eligible / blocked / in-progress / done）
- 直近の実装結果サマリ
- コードベースサマリ（ES-445 で v2 に導入予定のものを流用）

### B2. 横断メモリ

PM がタスクをまたいで知識を蓄積・参照する仕組み。

#### 2.1 メモリスコープ（最初から全部）

| カテゴリ | 内容 | 例 |
|---------|------|-----|
| PM 判断履歴 | 過去の選別理由・整理判断・失敗パターン | 「ES-XXX を優先した理由: ...」「分割判断: ...」 |
| 実装結果 | 成功/失敗/コスト/CI結果のサマリ | 「ES-XXX: merged, $0.52, CI初回通過」 |
| プロダクト知識 | ドメイン知識・ユーザーフィードバック・運用インシデント | 「認証モジュールは法的要件で...」「本番で発生した障害: ...」 |

#### 2.2 ストレージ

- 正本: 対象リポの `docs/memory/` にマークダウン
- B1 の `docs/specs/` と同じ思想（リポ正本・git 監査・API 不要）
- カテゴリ別ファイルで管理（肥大時はトランケーション / アーカイブ）

#### 2.3 更新経路

- A3 アクション列と同じ設計: Codex が `update_memory` アクションを出力 → オーケストレーターが検証・書き込み
- IPI 対策が A3 と同じ箇所に集約される
- commit/PR フローに乗せることで人間レビューも可能

#### 2.4 参照タイミング

- GROOM: 盤面整理の判断に過去の知識を活用
- SELECT: 選別精度の向上（v2 の A1 を強化）
- PLAN: brief 生成時にドメイン知識を反映

## 2. v2 との関係

| v2 の資産 | v3 での活用 |
|-----------|-----------|
| A1 SELECT（PM選別） | GROOM の後に実行。メモリ参照で選別精度向上 |
| A2 PLAN（brief生成） | メモリからドメイン知識を注入して brief の質向上 |
| B1 specs/（要求/要件定義） | 不変の錨として GROOM にも注入 |
| ES-445 コードベースサマリ | GROOM の入力にも流用 |
| CodexPlanner 基盤 | GROOM フェーズの Codex 起動に再利用 |

## 3. v2.5 との関係

v2.5（マルチ実行者レイヤ）は IMPLEMENT フェーズの実行者振り分け。v3（PM 自律化）は上流の PM 行為の拡張。直交するため独立に開発可能。ただし GROOM で PM が「このチケットは Tier 3 向き」と判断してラベルを付けるなど、v2.5 のルーティングと連携する接点はある。

## 4. アーキテクチャ概要

```
                    ┌─────────────────────────┐
                    │     Codex (PM)           │
                    │  GROOM: アクション列出力  │
                    │  SELECT: 1件選定          │
                    │  PLAN: brief 生成         │
                    └──────────┬──────────────┘
                               │ JSON アクション列
                    ┌──────────▼──────────────┐
                    │   Orchestrator           │
                    │  - アクション検証         │
                    │  - Linear API 代理実行    │
                    │  - docs/memory/ 書き込み  │
                    │  - 監査ログ記録           │
                    └──────────┬──────────────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                ▼
         Linear API      docs/memory/      SQLite DB
        (チケット操作)    (横断メモリ)      (監査ログ)
```

## 5. スコープ外（v4+）

- C2. Web ダッシュボード
- C3. サーバレス（GitHub Actions）化
- D1. 並列実行
- D2. 複数リポジトリ
- D3. タスク内セッション再開
- D4. Linear 以外のタスク源
